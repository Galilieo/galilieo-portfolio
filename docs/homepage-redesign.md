## 品牌命名与内容边界

后续正式网站名称采用 **Galilieo Atlas**。`Atlas` 表示由个人长期维护的项目、笔记与探索记录集合，不要求所有项目使用岛屿、海洋或星球命名，也不建立强制的世界观分类。

```text
正式名称：Galilieo Atlas
网站定位：Galilieo 的个人作品、开发笔记与探索记录
品牌短句：Works · Notes · Explorations
```

- `Galilieo Atlas` 是个人网站和内容容器；具体项目继续使用各自真实、独立的名称。
- `heart-island 心屿` 是当前重点项目和首页重要视觉坐标，不等同于整个个人品牌，也不约束未来项目命名。
- Header 与浏览器标题使用正式名称；Hero 仍需直接说明“AI 应用开发者 / 软件工程学生”，不能只依赖品牌氛围表达身份。
- SEO 与 README 可以使用更完整的事实描述，但不使用“技术实验室”“数字实验空间”等容易显得机构化或超出个人项目规模的措辞。
- 正式代码、SEO、RSS、README 与个人网站项目条目已统一使用 `Galilieo Atlas`；作者姓名仍为 `Galilieo`。

## 产品目标

改版后，访客应能在首屏快速回答：

1. Galilieo 是谁，以及当前技术方向是什么。
2. 正在做什么项目、参与什么实践。
3. 为什么 heart-island 心屿值得重点展示。
4. 从哪里查看项目、文章、归档和联系方式。

视觉要有明确的 Heart Island 识别，但不能把个人作品集变成纯装饰性概念站。内容层级、链接可达性、移动端阅读和加载稳定性高于特效数量。

## 已确认的区块设计

### Header

- 品牌使用 Galilieo Atlas；导航包含首页、项目、博客、归档、关于和主题切换。当前栏目使用强调色文字、略高字重和居中短线，并通过 `aria-current="page"` 提供语义状态；不使用胶囊包裹。
- Header 由所有页面共用 `SiteHeader.astro`，使用独立的 `--header-page` 与共享材质 token；页面内容宽度和 `body:has(...)` 环境样式不得改变 Header 外观。
- 桌面端保持紧凑横向排列；移动端保留可键盘操作的折叠菜单和无 JavaScript 静态导航。
- 滚动后允许轻微背景、边框和模糊，但 Header 不应过厚。

### 第一行：个人档案与功能卡

左侧个人档案卡按用户手稿固定为上下两段：

1. 上段为头像、个人名 Galilieo、身份和简短介绍；头像在左，文字在右。`Galilieo Atlas` 只用于站点品牌，不能作为个人姓名。
2. 下段左侧显示由 Content Collections 计算的 Works / Notes 数量，右侧以图标展示 GitHub、邮件和 Heart Island 联系资料。

个人档案卡的主体、留白和“查看完整档案”进入 `/about/`；底部 Works / Notes 分别进入项目与博客列表，GitHub、邮件和 Heart Island 图标保留独立目标。实现使用兄弟覆盖层而不是嵌套锚点，并为整卡、头像、统计和联系方式提供克制的位移、旋转或缩放反馈；Reduced Motion 下移除这些变换。卡片不再放“查看项目”和“阅读博客”大按钮，也不重复 Heart Island 介绍。

右侧为上下排列的 GitHub 与音乐卡：

- GitHub 卡当前接入官方 Public Events，以最近 30 天 × 7 类事件格呈现并缓存 30 分钟；文案始终使用“公开活动”，不冒充 Contributions。
- 音乐卡当前读取网易云歌单生成的静态清单，并与全站左下持久播放器双向同步；全站只有一个 `<audio>`，提供播放、暂停、进度、上下曲与紧凑歌单。它不自动播放，Astro 页面切换时持续播放，Reduced Motion 下不旋转封面。
- 时间、位置和天气位于底部状态条：静态回退为上海，浏览器端使用访客时区、IP 城市级粗定位和 Open-Meteo 当前天气；定位或天气失败时回退而不伪造数据。

### 第二行：主要项目与精选文章

- 左侧 Selected Works 为主要内容卡，默认展示 Heart Island，之后可切换其他真实项目；右侧 Selected Writing 优先展示 `featured` 公开文章，再以近期文章补足三篇。
- 两张卡统一使用全幅视觉、底部渐变信息层和圆点状态，不保留可见的前后按钮或页码。整张视觉卡进入详情，圆点位于链接外部，不能形成嵌套交互。
- 项目和文章在配置 `cover` 后优先显示真实封面；缺少封面时，Heart Island 使用受保护的现有 SVG，其他内容使用双主题 CSS 占位，不借用参考站素材。
- 两张卡自动轮播但不追求快速切换：项目与文章使用错开的约 7 秒周期，普通切换采用约 `320ms` 的低幅方向交叉淡入淡出。悬停、键盘焦点、页面进入后台或 `prefers-reduced-motion: reduce` 时停止自动切换；Reduced Motion 下手动选择立即生效，圆点和左右方向键保持可用。
- 项目和文章只读取真实 Content Collections；数量不足时减少轮播项并自然降级，不复制首项或创建虚构内容。

### 扩展区与状态条

- 第二行和状态条之间保留复用同一栅格的扩展能力：未来可以增加两列中卡或三列小卡，但当前正式首页不渲染空占位。
- About 摘要、Archive 摘要或其他模块只有在确有首页价值时才进入扩展区；完整内容仍由独立页面承载。
- 删除“全部项目 / 所有文章 / 时间归档 / 完整 About”横向文字导航行，避免与 Header 和卡片入口重复。
- 最底部状态条只承载时间、城市级粗略位置/天气和其他轻状态；无 JavaScript 时显示上海静态回退，网络失败时显示明确不可用状态，不显示虚构实时值。

## 组件策略

- 实施时先审计并复用当前 Astro 组件、Content Collections 查询和主题 token，再决定合并或替换哪些首页组件。
- 个人档案、功能卡组、项目轮播、笔记轮播和状态条各自保持单一职责；不创建无实际复用价值的通用 `GlassPanel`。
- 轮播使用完整服务端 HTML 输出和少量原生 TypeScript 渐进增强；无 JavaScript 时至少显示首项和可用详情链接，圆点控制不伪装成可用功能。
- 下一版验收后才归档不再引用的 `HeroVisual`、Quick Overview、Archive Preview 或 About Preview 等旧实现，不能在迁移过程中同时长期维护两套首页。
- 不引入 React、Vue、Svelte、Tailwind、GSAP、Three.js 或新的轮播运行时。

## 技术栈与扩展边界

当前真实技术栈为：

- Astro 7、TypeScript strict、Content Collections。
- Markdown / MDX、原生 CSS、CSS Variables、内联 SVG。
- 少量浏览器 TypeScript，以及由 `ClientRouter` 提供的 Astro View Transitions。
- RSS、Sitemap、ESLint、Astro Check、Prettier。
- pnpm 构建与 Nginx 静态部署。

允许在需求成立后评估两项轻量扩展，但它们不是当前能力，也不属于本次首页结构重构的默认范围：

- **Pagefind**：当公开文章和项目数量增长到导航、标签与归档不足以定位内容时，用于构建后静态搜索。接入前需确认中文分词效果、索引范围、构建产物大小和无 JavaScript 回退；不提前放置无功能搜索框。
- **Giscus**：当确实需要公开文章评论，且站点所有者愿意启用 GitHub Discussions、接受第三方请求与 GitHub 登录依赖时再接入。接入前需确认仓库、分类、主题同步、隐私说明、内容审核和加载失败回退；不用于项目数据或身份系统。

这两项都不得成为首页主要内容的渲染依赖，也不构成引入 React、Vue 或其他应用框架的理由。

### 第二层：复杂交互出现后使用 Vue 3

当单个功能已经出现多个联动状态、复杂键盘交互、可复用状态逻辑或明显高于原生 TypeScript 的维护成本时，可以单独评估 Vue 3。Astro 官方 [`@astrojs/vue`](https://docs.astro.build/en/guides/integrations-guide/vue/) 集成支持渲染和客户端激活 Vue 3 组件；它只用于局部交互岛，不把整站改造成 Vue SPA。

适合进入评估的组件包括：

- 项目多条件筛选。
- 博客即时搜索界面；搜索索引仍可由 Pagefind 提供。
- 命令面板。
- 需要明显超出现有原生播放器范围的复杂媒体编辑或多源队列工具；当前全站播放器继续使用已有原生 TypeScript Module。
- 复杂交互时间线。
- 多个状态联动的小组件。
- 已经单独批准 Three.js 场景后的控制面板；控制面板不能反过来成为引入 Three.js 的理由。

引入时遵守以下边界：

- 先证明原生 TypeScript 方案会产生明显状态耦合或维护负担；简单切换、单条件筛选和普通轮播继续使用 Astro 与原生脚本。
- Vue 组件直接从 `.astro` 页面接收构建期数据，优先保持组件局部状态；没有跨岛共享状态前不引入 Pinia 或全局应用入口。
- 按交互优先级使用 Astro 的客户端指令：首屏关键交互才考虑 `client:load`，低优先级使用 `client:idle`，下方或较重组件优先 `client:visible`，仅特定视口需要时可用 `client:media`。指令行为以 [Astro 官方参考](https://docs.astro.build/en/reference/directives-reference/#client-directives) 为准。
- 默认保留服务端生成的 HTML；除非功能无法服务端呈现且已有明确 fallback，不使用 `client:only`。
- 每个 Vue 岛都要单独检查脚本体积、加载时机、键盘操作、Reduced Motion、无 JavaScript 回退和 Astro 页面切换后的状态与清理。
- 不建立 Vue Router、全站 Vue 根节点或平行的 Vue 页面体系；Astro 文件路由、Content Collections、主题 token 和 Nginx 静态部署继续保持主架构。

## 内容与数据来源

- 身份、域名、导航、联系方式和外链继续由 `src/config/site.ts` 管理。
- 项目和文章继续来自 Content Collections。
- 项目数、公开文章数等可以构建时计算，不手写容易过期的统计数字。
- 首页固定状态与经历继续集中在现有 `src/data/home.ts`；迁移后删除不再消费的数据字段，不拆成多个只有几行的数据文件。
- `private: true` 当前是“项目内容不公开”的展示语义，不阻止项目说明页生成；页面不得泄露未公开业务细节。
- `cover` schema 已接入首页项目与文章轮播：真实封面使用 Astro 图片管线，未提供时回退到现有 Heart Island SVG 或统一 CSS 主题占位。当前内容仍未配置正式封面，因此占位不是最终素材。

## 双主题设计契约

日间和夜间都是正式展示状态，使用同一套 DOM、组件和语义 token，不复制两份布局或维护两套样式表。日间变量定义在 `:root`，夜间变量定义在 `html[data-theme='dark']`；现有系统偏好和本地持久化逻辑继续决定初始主题。

双主题色调已经固定：日间是暖骨白、雾蓝灰与低饱和石板蓝，夜间是深海蓝黑、靛蓝灰与少量月光紫蓝。它们不是彼此反色，也不是普通 SaaS 的白底蓝按钮与纯黑霓虹组合。

表中的弱文字、日间强调色与夜间强调前景已经针对小字号文字和按钮对比度校准，不要直接换回更浅的参考值。完整环境色还包括天空、海面、雾、岩石、植被和暖灯，唯一代码来源是 `src/styles/tokens.css` 的 `--home-*`；组件和页面不得另写主题专用十六进制颜色。

强调色只用于当前导航、主按钮、链接、时间轴节点、少量标签、岛屿核心光源和交互状态。中性色与蓝灰层次应占绝大多数；不要把每个标题、图标和边框都染蓝。白天不得用纯白作为页面底色，夜间不得用纯黑作为页面底色，暖黄色只能作为极少量灯光点缀。

阴影、场景叠层、岛屿环境光和 `theme-color` 必须随同一主题切换。开发时优先修改已有语义变量并同时验证两套主题、正文对比度和交互状态，不在面板或组件里追加临时颜色覆盖。

最终背景不是本阶段结构验收条件。背景资源、主题遮罩和内容面板必须解耦；未来替换 `src/assets/` 中的背景时不得重排仪表盘结构。

## 响应式

- `≥ 1000px`：两行仪表盘均采用约 `60:40`；首行右侧上下堆叠 GitHub 与音乐卡，第二行右侧为笔记轮播。
- `< 1000px`：不继续压缩右栏，按个人档案、GitHub、音乐、项目轮播、笔记轮播、状态条的内容优先级降为单列或局部两列。
- `< 768px`：所有主要卡片单列；联系图标与轮播圆点保持触控尺寸；状态条允许换行，不依赖横向滚动才能读取核心信息；减少大面积 `backdrop-filter`。

具体断点应和当前 1180px、920px、680px、580px 规则一起整理，避免保留两套相邻断点。移动端不能只是桌面图等比缩小。

## 动效与素材边界

P0 只保留必要的主题切换、菜单和低成本显现。P1 再增加卡片 hover 和页面进入动画。P2 才评估 3D 岛屿、动态海面、鼠标视差、光点、页面转场和环境声音。

任何阶段都必须：

- 保留 Reduced Motion。
- 页面离屏或后台时暂停持续绘制。
- 无 JavaScript 时通过 `html:not(.js)` 回退直接显示 Reveal 内容，并在窄屏提供静态导航。
- 不用高级视觉替代项目封面、文章摘要或导航信息。
- 不把参考图中的岛屿、头像和封面直接当成正式素材。
